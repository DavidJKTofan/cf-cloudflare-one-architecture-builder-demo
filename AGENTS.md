# AGENTS.md — Coding Agent Instructions

This repository is a **Cloudflare One Architecture Builder** — an interactive visual
walkthrough deployed as a Cloudflare Worker serving static frontend assets.

## Build / Dev / Test Commands

```bash
npm install                # Install dependencies
npm run dev                # Local dev server (wrangler dev)
npm run check              # TypeScript type-check (tsc --noEmit)
npm run types              # Regenerate Env types from wrangler.jsonc
npm run test               # Run all tests (vitest)
npx vitest run test/index.spec.js          # Run a single test file
npx vitest run -t "responds with"          # Run tests matching a name pattern
npm run deploy             # Deploy to Cloudflare Workers
```

The Worker (`src/index.ts`) is TypeScript. The frontend (`public/js/`) is vanilla
JavaScript with no build step — changes are live on save during `npm run dev`.

## Project Layout

```
src/index.ts                  # Cloudflare Worker entry point (TypeScript)
worker-configuration.d.ts     # Auto-generated Env types (do NOT hand-edit)
public/
  index.html                  # SPA shell — script load order matters
  css/app.css                 # All styles (CSS custom properties, dark and light themes)
  js/
    app.js                    # Thin orchestrator — renderAll(), init
    data/                     # Pure data objects (no DOM). Edit to change content.
      components.js           # Infrastructure + user element definitions
      connectors.js           # 17 Cloudflare connectivity options
      templates.js            # Use case template presets
      achievements.js         # Gamification milestones
    engine/                   # Core logic (no event binding)
      state.js                # State object + mutations (add/remove/reset)
      connections.js          # SVG Bezier curve rendering
      progress.js             # Progress bar + achievement checking
    ui/                       # DOM events + rendering (one concern per file)
      toast.js, drag-drop.js, sidebar.js, detail-panel.js, canvas.js, onboarding.js, export-image.js
    presets/
      full-sase.js            # Full SASE architecture preset
```

## Code Style — TypeScript (Worker)

- **Formatting**: Prettier — tabs, single quotes, semicolons, 140 char line width.
- **Target**: ES2022, strict mode, `noUncheckedIndexedAccess: true`.
- **Env type**: Generated by `wrangler types` — never hand-write the `Env` interface.
  After changing `wrangler.jsonc` bindings, run `npm run types`.
- **Export pattern**: `export default { async fetch(request, env, ctx) { ... } } satisfies ExportedHandler<Env>;`
- **Error handling**: Explicit `try/catch` with structured JSON logging via `console.error(JSON.stringify({...}))`.
  Never use `ctx.passThroughOnException()`.
- **No global request state**: Never store request-scoped data in module-level variables.
- **Floating promises**: Every Promise must be `await`ed, `return`ed, or passed to `ctx.waitUntil()`.
- **Security**: Use `crypto.randomUUID()` / `crypto.getRandomValues()`, never `Math.random()` for security.

## Code Style — Frontend JavaScript

All frontend JS is **vanilla** — no frameworks, no bundler, no ES modules.

### Module Pattern

Every file follows this structure:

```js
"use strict";
window.App = window.App || {};

/** JSDoc block describing the file's purpose. */

// Private helpers — plain function declarations (file-scoped)
function helperFn() { ... }

// Public API — named function expressions on window.App
window.App.doSomething = function doSomething() {
    const { state, COMPONENTS, renderAll } = window.App;
    // ...
};
```

Dependencies are destructured from `window.App` at the **top of each function body**,
not at file scope. This ensures correct resolution regardless of script load order.

### Naming Conventions

| Pattern | Prefix | Examples |
|---|---|---|
| Event wiring | `setup*` | `setupDragAndDrop()`, `setupTemplates()` |
| DOM rendering | `render*` | `renderAll()`, `renderConnections()`, `renderDetailPanel()` |
| State mutations | verb | `addElement()`, `removeConnection()`, `resetState()` |
| Checks | `check*` | `checkAchievements()` |
| Data constants | `UPPER_CASE` | `COMPONENTS`, `CONNECTORS`, `TEMPLATES`, `ACHIEVEMENTS` |
| State | `state` | Single mutable object at `window.App.state` |

### Formatting

- **Indentation**: Tabs (matching `.editorconfig` and `.prettierrc`).
- **Strings**: Double quotes in frontend JS; single quotes in TypeScript.
- **Semicolons**: Always.
- **const/let**: Prefer `const`; use `let` only when mutation is needed. Never `var`.
- **Functions**: Arrow functions for callbacks/lambdas; named `function` for top-level.
- **HTML generation**: Template literals building `innerHTML` strings.
- **Comments**: JSDoc block at file top; `/* --- Section --- */` dividers within files.

### Adding Content

| To add a... | Edit | Also |
|---|---|---|
| Component | `data/components.js` | Add sidebar button in `index.html` |
| Connector | `data/connectors.js` | Add button in `index.html` + CSS var in `app.css` |
| Template | `data/templates.js` | Add button in `index.html` `#templatesGroup` |
| Achievement | `data/achievements.js` | Nothing else needed |
| Preset | New file in `presets/` | Add `<script>` tag before `app.js` |

### Script Load Order (in index.html)

Order matters — each layer depends on the previous:

1. `data/*` — Pure data, no dependencies
2. `engine/*` — Depends on data
3. `ui/*` — Depends on data + engine (`toast.js` must load before others that call `showToast`)
4. `presets/*` — Depends on engine + ui
5. `app.js` — Last; defines `renderAll()` and runs init on `DOMContentLoaded`

## Testing

Tests use **Vitest** with `@cloudflare/vitest-pool-workers`. Test files are `.js` in `test/`.
The vitest config (`vitest.config.js`) runs tests inside the Workers runtime using the
wrangler config for bindings. Current tests are scaffold stubs and may need updating.

## Key Files to Check After Changes

- After editing `wrangler.jsonc`: run `npm run types` then `npm run check`.
- After editing any `.ts` file: run `npm run check`.
- After editing `index.html` script tags: verify load order matches the dependency layers above.
- After editing `data/*.js`: no build needed; changes appear on next page load in dev.
